# Authentication Architecture

## Overview

PocketDev uses a flexible authentication system designed for both protected and unprotected deployment scenarios. It is built on [Laravel Fortify](https://jetstream.laravel.com/features/fortify.html) with a single-admin model: only one user can exist, created via the first-run setup wizard.

## Authentication Modes

### 1. Full Authentication (Recommended for Public Access)
- Email/password with enforced TOTP (2FA) via Fortify
- Recovery codes (single-use) generated by Fortify's `RecoveryCode` helper
- Single admin account stored in database
- Session-based authentication

### 2. Permanent Bypass ("Don't Ask Again")
- Stored in database: `auth_bypass_permanent = true`
- Setup wizard won't appear on startup
- Suitable for VPN/Tailscale protected deployments

### 3. Session Bypass (Temporary)
- Stored in session: `auth_bypass_session = true`
- Only lasts for the current browser session
- Setup wizard will reappear after session expires

## First-Run Setup Wizard

On a fresh deployment with no users, the wizard at `/setup` walks the admin through:

1. `setup.index` — choose between "set up auth" or "skip auth"
2. `setup.credentials` — enter name, email, password (validated `email:rfc`, `unique:users,email`, lowercased on save)
3. `setup.totp` — scan QR, enter a TOTP code to verify the authenticator is working
4. `setup.recovery` — save recovery codes, tick the acknowledgement box, submit to create the user

The user is only persisted on the final step (`confirmRecovery`), so an abandoned wizard leaves no residue beyond the session. Session ID is regenerated after login to prevent fixation.

All `/setup/*` routes are rate-limited (`throttle:30,1`) and gated by `User::count() === 0`.

## Security Settings

Once an admin exists, `/settings/security` provides a dashboard for:

- Change password (requires `current_password`, rotates session, invalidates other DB sessions)
- Regenerate recovery codes (requires `current_password`, staged — old codes remain valid until user acknowledges new ones)
- Reset TOTP (two-step: verify current password + TOTP, then confirm new TOTP; old TOTP remains active until acknowledgement)
- Log out other sessions (database session driver only)
- Disable authentication (requires `current_password` + current TOTP + explicit checkbox; wipes the user and enables permanent bypass)
- Add password (for passwordless accounts — currently unused since all accounts are password-based, but wired for future OAuth support)

All `/settings/security/*` mutation routes are `auth`-gated and `throttle:10,1`-rate-limited.

### Two-Phase Commits for Recovery Codes

Any action that rotates recovery codes (first-time 2FA enrollment, regenerate, reset TOTP) stashes the new codes in `session('pending_2fa_commit')` and redirects to `/settings/security/recovery-codes`. The user sees the codes, must explicitly acknowledge them, and only then is the user record updated. This prevents account lockouts where 2FA is enabled but the user never saw their codes.

## Middleware: EnsureSetupComplete

The `EnsureSetupComplete` middleware (`app/Http/Middleware/EnsureSetupComplete.php`) protects both web and API routes.

### Check Order

1. **Public routes** — exact-prefix patterns, always accessible
2. **Permanent bypass** — `auth_bypass_permanent = true` in database
3. **Session bypass** — `auth_bypass_session` in session
4. **No users yet** — redirect to setup wizard
5. **Setup incomplete** — redirect to login (unauth) or provider setup (auth)
6. **Unauthenticated** — redirect to login

### Public Routes

```php
$publicRoutes = [
    'setup',                // First-run index
    'setup/skip',           // "Skip auth" handler
    'setup/credentials',    // Wizard step 1 (GET+POST)
    'setup/totp',           // Wizard step 2 (GET+POST)
    'setup/recovery',       // Wizard step 3 (GET+POST)
    'login',
    'logout',
    'two-factor-challenge', 'two-factor-challenge/*',
    'claude/auth', 'claude/auth/*',
    'codex/auth', 'codex/auth/*',
];
```

Setup routes are listed explicitly (not via `setup/*` glob) so that `/setup/provider` stays behind the normal auth flow. Adding a new first-run wizard step requires adding it here. Routes inside public prefixes that require auth (e.g. `POST /claude/auth/upload`) apply `auth` middleware per-route.

## API Routes Protection

### Why API Routes Are Web Routes

PocketDev's `/api/*` routes are **internal browser endpoints** (AJAX calls from the SPA), not an external API. They are defined in `routes/web.php` under a `Route::prefix('api')` group, which means they receive full web middleware automatically: session management, CSRF protection, cookie encryption, and the `EnsureSetupComplete` auth check.

### CSRF

All `/api/*` mutations require a valid CSRF token. The frontend fetch wrapper in `resources/js/bootstrap.js` auto-injects `X-CSRF-TOKEN` on same-origin `POST`/`PUT`/`PATCH`/`DELETE` requests (reading from the `<meta name="csrf-token">` tag included in every authenticated layout). Every form includes `@csrf`.

### Implementation

In `routes/web.php`, all API routes are grouped under the `/api` prefix:

```php
Route::prefix('api')->group(function () {
    // All AJAX endpoints (conversations, agents, settings, etc.)
});
```

There is no `routes/api.php` file. Since all endpoints are browser-only, there is no reason to use Laravel's API middleware group (which lacks session and CSRF support).

## Fortify Configuration

`Fortify::ignoreRoutes()` is called in `FortifyServiceProvider::register()` to disable **all** default Fortify route registration. Only the minimal set of routes needed for login/logout/2FA challenge are registered manually in `routes/web.php`:

```php
Route::get('/login', ...)->middleware('guest:web');
Route::post('/login', ...)->middleware(['guest:web', 'throttle:login']);
Route::post('/logout', ...)->middleware('auth:web');
Route::get('/two-factor-challenge', ...)->middleware('guest:web');
Route::post('/two-factor-challenge', ...)->middleware(['guest:web', 'throttle:two-factor']);
```

Fortify's built-in 2FA management endpoints (`/user/two-factor-*`), password confirmation, registration, and all other routes are completely disabled — the single-admin model and custom `SecuritySettingsController` flows replace them entirely.

`config/fortify.php` enables only `twoFactorAuthentication` (with `confirm: true`). Registration, password reset, email verification, profile updates, and password updates are all disabled.

Rate limiters in `FortifyServiceProvider`:

- `login` — 5/min per (username + IP) combination
- `two-factor` — 5/min per `login.id` in session, falling back to IP if the session key isn't set

## Database Settings

| Key | Type | Description |
|-----|------|-------------|
| `auth_bypass_permanent` | boolean | Permanent auth bypass (user chose "don't ask again") |
| `setup_complete` | boolean | Whether provider setup wizard was completed |

## Session Keys

| Key | Description |
|-----|-------------|
| `auth_bypass_session` | Temporary auth bypass (current session only) |
| `setup.pending_user` | First-run wizard: hashed name/email/password between wizard steps |
| `setup.totp_secret` | First-run wizard: TOTP secret between wizard steps |
| `setup.recovery_codes` | First-run wizard: generated recovery codes shown before user creation |
| `pending_2fa_commit` | Pending recovery-code acknowledgement (add-password / regenerate / reset-totp) |
| `add_password.*` / `reset_totp.*` | Step-local state for multi-step security flows |

## Security Considerations

1. **Network-level security**: For private deployments behind VPN/Tailscale, auth bypass is acceptable. The permanent bypass is a one-way door triggered only by the admin (or, on fresh install, any visitor during the first-run race — see below).
2. **Public internet**: Always use full authentication with TOTP.
3. **API protection**: All API routes are protected identically to web routes (CSRF + session + EnsureSetupComplete).
4. **First-run race**: On a fresh install with no user, any visitor to `/setup` can complete the wizard or permanently disable auth. Mitigate by performing first-run setup before exposing the instance, or binding the deployment to `127.0.0.1` until setup is complete.
5. **Sensitive actions require friction**: change password, regenerate recovery codes, reset TOTP, disable auth all require `current_password`. Disable auth additionally requires a current TOTP code and an explicit confirmation checkbox.
6. **Session fixation**: session ID is regenerated after login, after wizard completion, after password change, and after enabling 2FA.
7. **Other-session invalidation**: changing the password invalidates all other sessions for the user (database session driver only).
